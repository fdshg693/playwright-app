## セッションサーバー（scripts/server）

SPEC.mdの「ブラウザセッションは永続、AIコンテキストはステップごとに初期化」を担うFastAPIサーバー。Step2（骨組み・人間駆動のsnapshot/command）とStep3（`/run`によるAI駆動の全自動実行）の実装が同居する。責務は[.claude/plan/main/02-server-skeleton.md](../plan/main/02-server-skeleton.md)・[.claude/plan/main/03-task-orchestration.md](../plan/main/03-task-orchestration.md)に詳しい。

- `session_manager.py` の `SessionManager` が `session_id -> CliExecutor` のin-memoryレジストリ。`session_id` はplaywright-cliの `-s=` セッション名そのもの。`POST /sessions`で受け取る`story`は**ストーリーYAMLファイルへのパス**として`create()`内で即座に`story.load_story()`によりパースされ、`Story | None`として保持される（`StartSessionRequest.story: str`という外部契約自体は生文字列のまま）。並行アクセスの安全性は未対応（ロックなし）。
- `app.py` が4+1本のHTTPエンドポイントを持つ。`/snapshot`・`/command`はStep2由来で人間が直接叩く想定、`/run`はStep3で追加した自動実行エンドポイント。OpenAIクライアントは`_get_client()`でモジュール内シングルトンとして遅延初期化する（`config.get_api_key()`が`OPENAI_API_KEY`未設定で例外を投げるため、トップレベル生成するとAI不要な既存エンドポイントまで起動不能になる）。`POST /sessions`は`cli.open(target_url)`のActionResultを`sessions.set_seed_code()`でセッションに紐付けて保存する（下記の`seed_code`参照）。
- `orchestrator.py` の `run_story(cli, client, model, story, out_path, seed_code=None)` が`/run`の実体。**新しいAIループは書いていない** — `scripts/vertical_slice/runner.run_steps()`（内部で`step_runner.run_step`をステップごとに呼ぶ）と`runner.write_and_test()`（`write_spec_file`/`write_failure_notes`/`run_playwright_test`をまとめたもの）をそのまま再利用している。`runner.run_vertical_slice()`自体は使わない（`story.seed_url`へ毎回navigationする前提のため。サーバーでは`POST /sessions`の`target_url`で既にnavigation済みなので二重navigationを避けている）。タスク（＝ステップ）の完了判定もStep1の`finish_step`宣言方式をそのまま踏襲し、サーバー側で独自のチェックポイント判定は行わない。`write_spec_file`はStep4でステップ境界つきの`runner.StepBlock`リストを受け取るシグネチャに変わったため、`run_story`は各ステップの`step_code`を`generated_code`へ単純extendするのではなく`StepBlock(step=step, code=step_code)`として積んでいく。Step5で`run_steps`/`write_and_test`が`<out>.tasks.jsonl`記録込みの共有関数として`runner.py`に切り出されたため、`run_story`自体はseed_codeの扱いだけを担う薄いラッパーになっている（[[vertical-slice-runner]]参照）。`run_story`/`resume_story`は先頭で`scripts/vertical_slice/run_id.new_run_id()`を発行し、`runner.run_steps()`にそのまま引き回す（`.steps.jsonl`/`.tasks.jsonl`のパス決定はrunner側の責務、[[vertical-slice-runner]]参照）。戻り値は`(passed, spec_path, failure_notes, run_id)`の4-tupleで、`app.py`が`RunResponse`/`ResumeResponse`の`run_id`フィールドへそのまま渡す。旧来の`unlink()`による毎回上書きは廃止し、実行のたびに新規ファイルへ書く。
- `seed_code`（`run_story`の引数）は`POST /sessions`時点の`cli.open(target_url)`が生成した`page.goto(...)`相当のコードで、`story.steps`をいくら辿っても復元できない（ストーリーの番号付きステップではないため）。`app.py`の`/run`ハンドラが`sessions.get_seed_code(session_id)`で取り出して渡し、`run_story`は非空なら`StepBlock(step=None, code=[seed_code])`として`blocks`の先頭に積む。**Step4の実AI検証（`scripts/temp/test_server_ai_run.py`）で発見した不具合の修正**: 元々`run_story`はこの値を一切保存・使用しておらず、`/run`で組み立てた`.spec.ts`に`page.goto`が無いまま出力されていた（サーバー内の`CliExecutor`は`POST /sessions`で既にnavigation済みなので`/run`自体は動くが、生成された`.spec.ts`を単独で`npx playwright test`にかけると空白ページから始まりタイムアウトする）。「AIコンテキストはステップごとに初期化されるが、生成物は独立して再実行可能」というSPEC.mdの前提を壊す不具合だったため、Step3のファイルではあるがStep4の一環で修正した。
- `/run`はHTTPリクエスト内で同期的に最後まで実行する。進捗のストリーミング配信や途中中断のAPIは無い（Step7スコープ）。生成物の出力パスは`story.name`ベース（`tests/generated/{story.name}.spec.ts`）で、`session_id`は使わない。同じストーリーを複数セッションで並行実行すると出力ファイルが競合するが、これも未対応。
- `main.py`は起動時に`load_dotenv()`を呼ぶ（`.env`の`OPENAI_API_KEY`を`/run`のOpenAIクライアントが読めるようにするため）。
- `/run`は実AI APIへの課金コールを伴う。人間が直接叩く分には確認不要だが、動作確認としてこのエンドポイントをスクリプトから叩く場合は[[vertical-slice-runner]]や`vertical-slice-ai-test`スキルと同様「実行前にユーザーへ確認」の方針を踏襲する（[architecture.md](architecture.md)参照）。`scripts/temp/test_server_ai_run.py`が`custom-pages-demo.yaml`に対して`/sessions`→`/run`→`/sessions/{id}`(DELETE)を実行する再現可能なスモークテスト（`scripts/temp/test_server.py`の`/run`版）。`vertical-slice-ai-test`スキルは`scripts/vertical_slice/main.py`経由の実行しかカバーしないため、サーバー経由（Step3の`/run`層そのもの）を検証したい場合はこちらを使う。
- `POST /sessions/resume`（Step5）が途中再開・分岐実行のエントリ。ボディは`ResumeRequest`（`tasks_log: str`, `resume_before_step: int`, `story: str`）。`sessions.create(session_id, story=body.story)`でnavigationなしの新セッションを作り（`SessionManager.create()`自体がnavigationしないのはStep3から変わらない）、`orchestrator.resume_story(cli, client, model, story, out_path, tasks_log, resume_before_step)`を呼ぶ。`resume_story`は`runner.build_resume_state()`（`tasks_log`から早送り用コードと`.spec.ts`用`prior_blocks`を組み立てる、実体はCLI版`runner.resume_vertical_slice`と共有）→ `cli.open()` + `cli.run_code(replay_source)`で早送り → `runner.run_steps`/`write_and_test`で`story.steps`を通常実行、という流れ（[[vertical-slice-runner]]の「記録と途中再開・分岐」参照）。`story`は`tasks_log`を生成した元セッションのストーリーと別YAMLでもよく、分岐実行はこれで表現する。レスポンスは`ResumeResponse`（`session_id`/`passed`/`spec_path`/`failure_notes`/`run_id`）。`/sessions`同様、成功後にセッションを自動closeはしない。
