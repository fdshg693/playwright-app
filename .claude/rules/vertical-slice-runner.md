## vertical_slice ランナー

SPEC.md の Step1（縦の一本通し）実装。`python -m scripts.vertical_slice.main --story <yaml>` で、人間の介入なしにストーリー1本を最初から最後まで実行し、`tests/generated/*.spec.ts` を生成する。責務は `.claude/plan/main/01-vertical-slice.md` に詳しい。

- `story.py` — `scripts/stories/*.yaml`（`seed_url` + `steps[].instruction` + `intent`）を `Story`/`Step` にロードするだけ。`intent`はこのシナリオが何を検証したい・できるかを書く必須の自由記述フィールドで、実行時のロジックには一切使われない（生成される`.spec.ts`にも反映しない）人間・AI読者向けドキュメンテーション専用。欠けていると`load_story`が`KeyError`で落ちる。**`steps[].instruction`は必ず1行で書く**（`intent`と違い改行可能ではない）。後述の`write_spec_file`が`instruction`を生の改行エスケープなしでコメント行に埋め込むため、YAMLの`|`等で改行を含めると生成`.spec.ts`がTypeScript構文エラーになる。story YAML執筆時の書式・落とし穴は [scripts/stories/README.md](../../scripts/stories/README.md) にまとめてある。
- `cli_executor.py` の `CliExecutor` が SPEC.md でいう「サーバー」役。ネットワークサーバーではなく、1つの名前付き playwright-cli セッション（`-s=<session>`）を最初から最後まで維持するプロセス境界。**playwright-cli はコマンド失敗時も exit code 0 のまま stdout に `### Error` を返す**ため、成功判定は exit code ではなく毎回 stdout をこの文字列でチェックしている（`CliExecutor.execute`）。`snapshot_text()`（`snapshot --json`）は`### Error`ではなく`isError: true`というJSON形状で異常を返すことがある（ファイルチューザーモーダルが開いている時など）。これも同じ`CliError`に揃えて投げる（Step6、実AI呼び出しテストで発見・修正。以前は無条件で`["snapshot"]`アクセスしており未捕捉の`KeyError`でプロセスがクラッシュしていた）。
  - **Step8（安全性ガードレール、URL許可リスト）**: コンストラクタが`allowed_domains: list[str] | None = None`を受け取り`self._allowed_domains`として保持する（`None`＝制限なし）。`execute()`が`### Error`チェックの直後・`ActionResult`構築の前に、playwright-cliが**あらゆる**コマンド実行後に返す`### Page\n- Page URL: <url>`行（`.claude/skills/playwright-cli/SKILL.md`「Snapshots」節）を`_PAGE_URL_RE`で抜き出し、`urllib.parse.urlparse(url).hostname`（ポート番号を含まない・小文字化済み）が`allowed_domains`（`_host_allowed`、完全一致または`*.`接頭辞のサブドメインワイルドカード）に無ければ新設の`DisallowedNavigationError`（`CliError`のサブクラス）を投げる。`about:blank`／空文字はチェック対象外として常に許可する。この1箇所への集約により、`navigate`ツール呼び出し・クリックが誘発する間接遷移・`run-code`によるリプレイ・`/command`のすべてを同じ関門でカバーする（`generate_locator`/`eval_raw`は`--raw`/`--json`でPage節自体が出力に含まれないため`execute()`を経由せず、このチェックの対象外）。
- `prompts.py` の `build_input()` が「1ステップ＝1フレッシュコンテキスト」の起点。呼び出しごとに `previous_response_id` は使わず、developer prompt + 残りステップ + 現在のsnapshot からゼロで組み立てる。ステップ内の複数ターンは `step_runner.run_step()` がこの `input_items` リストにローカルで追記して繋ぐ（＝ステップを跨いだ記憶は持たないが、ステップ内はチェーンする）。
- `tools.py`/`step_runner.py`（Step8）: `step_runner.run_step()`の`tools.execute_tool(...)`呼び出しの`try`に、既存の`except CliError`より前に`except DisallowedNavigationError as exc:`を追加した。`failure_notes`へ既存の`cli_error`分岐と同じフィールド構成（`step`/`reason`/`tool`/`arguments`/`error`）で`reason: "disallowed_url"`を積み、`stop = True`・`stop_reason = "disallowed_url"`にする（`tools.py`自体の変更は無し。`DisallowedNavigationError`は`cli_executor.execute()`が投げ、`tools.execute_tool`はそれをそのまま伝播させるだけ）。
- `tools.py` の `TOOL_SCHEMAS` が OpenAI Responses API に渡すツール定義一式。`finish_step` はCLIコマンドではなくループ制御用の合図で、**他の操作系ツールと同じターンで一緒に呼んではいけない**契約（呼ばれた場合 `step_runner.run_step()` は同時に来た操作呼び出しを無視してログに warning を出す）。操作系ツールは `ref`（snapshotに載っているものだけ）でしか要素を指定できない。`add_expectation`（`ref`/`matcher`(`toBeVisible`|`toHaveText`)/`description`）も他の操作系ツールと同じ扱いの読み取り専用ツールで、確認のみのステップで使う。AIには`eval`スクリプトやロケータ文字列そのものを書かせず、`cli_executor.generate_locator`/`eval_raw`（`--raw`付きのCLI呼び出し）でこちら側が安定ロケータ・期待テキストを取得し、`await expect(page.<locator>).<matcher>(...)` 相当のTypeScript文を組み立てて返す（`tools._add_expectation`）。
- `step_runner.py` の `run_step()` が1ステップ分のマルチターン tool-calling ループ本体（前段落2つの実体）。ターンごとに `step_log.append_step_log()` を呼んで生ログを`<out>.steps.jsonl` に追記する。`run_step()`は`attempt: int = 1`引数を受け取り、`.steps.jsonl`の各エントリに`"attempt"`フィールド（後述のリトライ機構が何回目の試行かを渡す）が付く（Step6）。
- `runner.py` の `run_vertical_slice()` がストーリー全体のオーケストレーション（seed_url を開く → 各ステップで `step_runner.run_step()` を呼ぶ → 生成物を書き出す）と、生成物の書き出し（`write_spec_file` / `write_failure_notes`）・`npx playwright test` の実行（`run_playwright_test`）を担う。`run_steps()`（ストーリーのステップ列を先頭から実行して`StepBlock`/`failure_notes`を積む）・`run_task_logged_step()`（1ステップ分を`step_runner.run_step()`に委ねつつ前後のタスクログを取る）・`write_and_test()`（`.spec.ts`書き出し〜`npx playwright test`実行）は`run_vertical_slice`/`resume_vertical_slice`（後述）双方、および`scripts/server/orchestrator.py`の`run_story`/`resume_story`からも共有で呼ばれる（[[session-server]]参照）。`run_task_logged_step()`は`MAX_STEP_ATTEMPTS`（=3、初回1回＋リトライ2回）まで`run_step()`をフレッシュな試行として呼び直す（Step6）。リトライ中はブラウザ状態を明示的にロールバックしない（前の試行が実際に行った操作の結果を引き継いだまま次の試行に進む）。最終試行も失敗した場合にのみ`cli.console()`/`cli.requests()`を1回ずつ追加で取得し、その試行の`failure_notes`各エントリに`diagnostics`/`hint`を付与する。`before`/`after`のsnapshot/screenshot取得自体が`CliError`になった場合（ブラウザがモーダル等で壊れた状態から回復しないまま`MAX_STEP_ATTEMPTS`を使い切った場合）も、例外を伝播させず`"<... unavailable: ...>"`というプレースホルダ文字列に差し替えて記録を継続する。
- `run_steps()`/`run_task_logged_step()`は`should_stop: Callable[[], bool] | None = None`引数を受け取る（Step7、サーバーの`/sessions/{id}/stop`から`scripts/server/orchestrator.py`経由で渡される。詳細は[[session-server]]）。CLI（`run_vertical_slice`/`resume_vertical_slice`、`scripts/vertical_slice/main.py`）は`should_stop`を渡さずデフォルト`None`のままで、1プロセス1実行なのでCtrl+C（SIGINT）で代替する。停止の反映粒度は「`run_steps()`のステップ境界（次のステップに着手する直前）」と「`run_task_logged_step()`のリトライ試行境界（失敗したリトライ試行の直後）」の2箇所のみで、`step_runner.run_step()`のターン制御ループ（`MAX_TURNS_PER_STEP`）には一切手を入れない。ステップ境界での停止は`run_task_logged_step()`自体を一切呼ばずに`failure_notes`へ`reason: "stopped"`を1件積んでbreakする（＝そのステップは`<out>.tasks.jsonl`に記録されない）。リトライ境界での停止は性質が異なる: そのステップは実際に(部分的に)失敗しているので`reason`は最後の試行の実際の失敗理由（`cli_error`/`blocked`等）のまま変えず、代わりに`failure_notes`の各エントリへ`"stopped": true`フィールドを付ける（`attempt`と`MAX_STEP_ATTEMPTS`の比較から間接的に推測させない明示的なフラグ）。
- `FAILURE_REASON_HINTS`（Step8）に`"disallowed_url"`を追加した。`run_task_logged_step()`の試行for文は、直近の`step_failures`に`reason == "disallowed_url"`が含まれる場合、`should_stop`の早期脱出とは別に、残り試行を消費せず即座にbreakする（許可外ドメインに対してAIへ追加のツール呼び出しをさせ続けないため）。これはStep7の`stopped`/`"stopped": true`とは独立した別軸: `disallowed_url`は`reason`自体がリトライを打ち切る新しい打ち切り条件であり、`stopped_mid_retry`のような専用フラグは増やさない（`reason`が"disallowed_url"であること自体が既に理由を表しているため）。
- `step_log.py` が `<out>.steps.jsonl` の直列化・追記ロジック（`serialize_output_item` / `truncate` / `append_step_log`）を持つ。診断ログのフォーマットに関する責務はここに閉じる。`step_runner.run_step()`がターンごとに書き込む`model`フィールドは（Step5で）リクエスト文字列ではなくAPIレスポンスが実際にエコーバックした`response.model`に変更した。リクエスト時に指定した文字列は`requested_model`として別途残る（`cost_summary.py`のpricing lookupは`model`優先のまま変更不要）。`step_log_path(out_path, run_id)`は`{run_id}__{stem}.steps.jsonl`を`task_log.history_dir(out_path)`配下に書く（後述）。
- `task_log.py` が `<out>.tasks.jsonl`（タスク＝ステップ単位の記録）の直列化・読み込み・リプレイ用コード列の組み立てを持つ。`step_log.py`との責務分担: `.steps.jsonl`はAIターン単位の生ログ、`.tasks.jsonl`はタスク単位で完結した「実行前後のsnapshot/screenshot・そのタスクで生成されたコード列・成否」のみを持つ（`task_log_path` / `append_task_log` / `load_task_log` / `build_replay_source` / `recordings_dir` / `history_dir`）。screenshotの保存先は`recordings_dir(out_path)`（`{stem}.recordings/`、例: `tests/generated/search-demo.recordings/3-before.png`）。ログ本体（`.steps.jsonl`/`.tasks.jsonl`）の保存先は`history_dir(out_path)`（`{stem}.history/`）で、`tests/generated/`直下には増えない。`run_id.py`の`new_run_id()`（秒精度・ソート可能な`%Y%m%dT%H%M%S`文字列）が実行のたびに一意なファイル名プレフィックスを発行し、`{run_id}__{stem}.steps.jsonl`/`.tasks.jsonl`として実行ごとに新規ファイルへ書く。旧来の`unlink()`による毎回上書きは廃止した（過去実行の履歴が残る）。`run_id.parse_run_id_prefix(name)`がファイル名からプレフィックスを`datetime`へ復元する（`cost_summary.py`側の時間範囲絞り込みで使用予定、`run_id`プレフィックスの無い旧形式ログは`None`を返す）。
- `main.py` は argparse と各コンポーネントの配線（`CliExecutor` / `OpenAI` client / `load_story`）だけを行うCLIエントリポイント。実行ロジックは持たない。`--resume-tasks-log`/`--resume-before-step`を指定すると`run_vertical_slice`の代わりに`runner.resume_vertical_slice`を呼ぶ。両関数とも戻り値は`(passed: bool, run_id: str)`になり、`main.py`は`run_id`を`logger.info`で出力するだけ（呼び出し元がどのログファイルを見ればいいか特定できるようにするため）。

### 記録と途中再開・分岐（Step5）

- 復元方式は「記録済みの操作コード列を先頭から`playwright-cli run-code`で再実行する」。`state-save`/`state-load`（cookie/localStorageのみ復元、ナビゲーション履歴・DOM状態は復元不可）は使わない。`cli_executor.CliExecutor`に`screenshot(path) -> str`（`--filename=`は**呼び出し元プロセスのcwd基準**で解決され、親ディレクトリが無いとENOENTになるため`screenshot()`側で`mkdir(parents=True)`する）と`run_code(source) -> ActionResult`（`execute("run-code", [source])`）を追加した。`console() -> str`/`requests() -> str`（`execute("console"/"requests", ["--raw"]).raw_output.strip()`）も同様に薄いラップで、専用の生文字列パースは不要（Step6、失敗時診断用）。
- `run_code`は新しいセッションで`cli.open()`（URL無し、ブランクページを開くだけ）を呼んだ後でないと使えない（未openだと`playwright-cli`がエラーを返す）。`build_replay_source`が組み立てる`"async page => { ... }"`文字列には`await expect(...)`行を含めない（`run-code`のサンドボックスに`expect`は無い）。除外は早送り用の一時文字列のみに適用し、`.spec.ts`用の`prior_blocks`（`runner.build_resume_state`が組み立てる）にはassertion込みの元コードをそのまま使う。
- `runner.resume_vertical_slice(story, cli, client, model, out_path, tasks_log_path, resume_before_step)`が復元の実体。`build_resume_state()`で`tasks_log_path`から`replay_source`（早送り用）と`prior_blocks`（`.spec.ts`用、`step_id`が`None`または`resume_before_step`未満のタスクをそのまま`StepBlock`化したもの）を組み立て、`cli.open()` + `cli.run_code(replay_source)`で早送りしたのち、`story.steps`を（`resume_before_step`によるフィルタなしで）先頭から通常のAIループで実行する。**`story`は`tasks_log_path`を生成した元のストーリーと別のYAMLでもよい** — 「タスク3までは記録から復元し、タスク4以降を別の入力で実行する」という分岐実行は、分岐先のステップだけを持つ別YAML（`scripts/stories/search-demo-branch.yaml`が例。ステップIDは元ストーリーと不連続でもよい）を`story`に渡すことで実現する。`scripts/server/orchestrator.py`の`resume_story`も同じ`build_resume_state`を呼ぶだけの薄いラッパー（[[session-server]]参照）。`build_resume_state()`で読む`tasks_log_path`は再開元（過去の実行が発行した`run_id`のファイル）であり、この関数自身が発行する新規`run_id`（今回の実行分のログ）とは別ファイルなので、両者の読み書き順序に制約は無い。

### ステップ内ループの終了条件（`step_runner.run_step`）

`MAX_TURNS_PER_STEP = 8` の間、`finish_step` が呼ばれるまでターンを重ねる。以下のいずれかで即停止し `failure_notes` に理由が積まれる：`finish_step(status="blocked")` / モデルがツールを1つも呼ばない / CLI呼び出しが `CliError` を投げる / `MAX_TURNS_PER_STEP` 到達。これは**リトライ機構ではない**（リトライは Step6 の未実装スコープ、SPEC.md 6章）— 1ステップの応答時間に対する安全弁のみ。

### 生成物

- `<out>.spec.ts` — ステップ境界を保持した `runner.StepBlock` のリスト（`open()`分＋各ステップの生成コード）を1つの `test()` に組み立てて書き出し（`runner.write_spec_file`）。各ステップのコードの先頭には `// {step.id}. {step.instruction}` コメントが付く（test-generation.md 2.2節の規約。`open()`分にはステップ番号が無いのでコメントは付けない）。この埋め込みは単純なf-stringで改行のエスケープをしないため、`instruction`に生の改行が入っていると構文が壊れる（story YAML側の注意点。前述`story.py`の項および[scripts/stories/README.md](../../scripts/stories/README.md)参照）。書き出し後 `npx playwright test` を自動実行する（`runner.run_playwright_test`）。
- `<out>.failure-notes.json` — 失敗があった場合のみ（`runner.write_failure_notes`)。各エントリは（Step6）`attempt`/`max_attempts`（何回目/最大何回の試行だったか）と`hint`（`reason`の機械的な説明文、`runner.FAILURE_REASON_HINTS`）を持ち、最終試行の失敗分のみ`diagnostics`（`console`/`requests`/`snapshot`/`screenshot_path`）も持つ。（Step7）`FAILURE_REASON_HINTS`に`"stopped"`（未着手のまま終わったステップ、`run_steps()`のステップ境界で外部から停止要求が来た場合）が追加された。これはリトライ境界で停止した場合に各エントリへ付く`"stopped": true`フィールドとは別物: `reason: "stopped"`はそのステップ自体が一度も試行されなかったことを表し、`"stopped": true`は元の`reason`（`cli_error`/`blocked`等）はそのままに「リトライ途中で打ち切られた」ことだけを示す。（Step8）`reason: "disallowed_url"`で打ち切られたステップも、`run_task_logged_step()`自体は最後まで実行される（その試行は実際に走って失敗した）ため通常の失敗ステップと同様`<out>.tasks.jsonl`にエントリが作られる — `reason: "stopped"`（`run_task_logged_step()`自体が呼ばれず未着手）とは異なる。
- `{stem}.history/{run_id}__{stem}.steps.jsonl` — 全ターンの生ログ（プロンプト・モデル出力・ツール結果）。停止理由を後から読み返すためのもので、書式は `step_log.py` 冒頭のモジュールdocstringに説明がある。`run_id`（`run_id.new_run_id()`、秒精度）が実行ごとに発行され、実行のたびに新規ファイルとして残る（過去実行を`unlink`で消して上書きすることはしない）。
- `{stem}.history/{run_id}__{stem}.tasks.jsonl` — タスク（＝ステップ）単位の記録。1行1タスクで、実行前後のsnapshot・screenshotパス・そのタスクで生成されたコード列・成否を持つ（`task_log.py`）。途中再開・分岐実行のリプレイ元になる。同じ`run_id`が同一実行内の`.steps.jsonl`と対になる。（Step6）新規`"attempts"`フィールドを持つ: 1なら初回成功、2以上ならリトライが発生したことを示す。`code`はリトライがあった場合も全試行分を試行順に連結した1本のリストのまま（1ステップ＝1エントリの不変条件は保たれる）。（Step7）`run_steps()`のステップ境界で停止によって未着手のまま終わったステップは、`run_task_logged_step()`自体が呼ばれないためこのファイルにエントリが作られない。再開時は`resume_before_step`にそのステップのidをそのまま渡せばよい — `build_resume_state`/`build_replay_source`の既存フィルタ（`step_id < resume_before_step`）が自然に「未着手ステップは次回、通常のAIループで再実行される」という挙動になる。
- `{stem}.recordings/` — `<out>.tasks.jsonl`が参照するscreenshot（`{step_id}-before.png`/`{step_id}-after.png`、seedブロックは`seed-after.png`のみ）の保存先。`.history/`と違い`run_id`では分かれておらず、ファイル名は実行を跨いで固定（同じstepを再実行すると上書きされる）。今回のrun-scoped化のスコープ外。
