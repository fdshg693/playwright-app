# Step 8 詳細版: 最低限の安全策

> [big_plans/08-safety-guardrails.md](../../../big_plans/08-safety-guardrails.md) の詳細版。[00-overview.md](00-overview.md)参照。big_plans記載の3本柱（URL許可リスト・最大同時セッション数・アイドルタイムアウト）を実装する、ロードマップ最後のステップ。

## やること

- **URL許可リスト**: `scripts/vertical_slice/config.py`に`get_allowed_domains() -> list[str] | None`を追加（`ALLOWED_DOMAINS`環境変数、カンマ区切りホスト名、`*.example.com`形式のサブドメインワイルドカードを許容、未設定なら`None`＝制限なし）。実際のチェックは`CliExecutor.execute()`の中に1箇所だけ実装する。playwright-cliは**あらゆる**コマンドの実行後にその時点のページ状態を`### Page\n- Page URL: <url>\n- Page Title: ...`として返す（`.claude/skills/playwright-cli/SKILL.md`「Snapshots」節: "After each command, playwright-cli provides a snapshot of the current browser state."）ため、`execute()`が`out`から`Page URL:`行を正規表現で抜き出し、許可リストに無いホストなら新設の`DisallowedNavigationError`（`CliError`のサブクラス）を投げる、という単一の関門にする。ホスト比較は`urllib.parse.urlparse(url).hostname`（ポート番号を含まない）で行う——`scripts/stories/*.yaml`の`seed_url`はほとんどが`http://localhost:8080/...`であり、ポート込みで比較すると`ALLOWED_DOMAINS=localhost`という直感的な設定では通らなくなるため。これにより`navigate`ツール呼び出しだけでなく、クリックが引き起こす間接的な遷移・`run-code`によるリプレイ・`POST /sessions`の`target_url`・人間駆動の`/command`のすべてを1箇所の実装で一様にカバーする（`navigate`ツール／`target_url`だけをチェックする案は、クリックが誘発する遷移を取りこぼすため採らない）。
- 許可リスト外への遷移が起きた場合の挙動は**即停止**に決める（big_plansが開いたままにしていた論点）。`DisallowedNavigationError`は`CliError`を継承するので、既存の`except CliError`各所にそのまま拾われて即座にその場のコマンド実行を止める。加えて、AI駆動パス（`step_runner.run_step()`のツール呼び出しループ）では新しい`except DisallowedNavigationError`節を追加し、`failure_notes`に既存パターンと同じ形で`reason: "disallowed_url"`を積む（`runner.FAILURE_REASON_HINTS`に対応するヒント文言を追加）。人間駆動パス（`POST /sessions/{id}/command`）と`POST /sessions`の`target_url`では、HTTPレイヤーで`DisallowedNavigationError`を`CliError`より先に拾い、生成物としての記録は無いため代わりにHTTP 400（ポリシー起因の拒否。既存の502＝CLIレベルの実行時失敗とは意味的に区別する）で即座に伝える。
- `disallowed_url`はリトライ対象から除外する。`runner.run_task_logged_step()`の試行ループに、`step_failures`の`reason`が`disallowed_url`なら残り試行を消費せず即break、という条件をStep7の`should_stop`早期脱出と同じ位置に追加する（許可されていないドメインに対してAIに追加のツール呼び出しをさせ続けるのは「即停止」の趣旨に反するため）。
- **最大同時セッション数**: `config.py`に`get_max_sessions() -> int`（`MAX_CONCURRENT_SESSIONS`環境変数、デフォルト`5`）を追加。`SessionManager`はコンストラクタで`max_sessions: int`を受け取り、`create()`の先頭で現在のセッション数と比較し、上限到達時は新設の`SessionLimitExceededError`を投げて拒否する（CLIExecutorのプロセスは起動しない＝チェックはプロセス起動より前）。`app.py`の`POST /sessions`（`start_session`）・`POST /sessions/resume`（`resume_session`）は両方とも`sessions.create()`を呼ぶため、両方に`except SessionLimitExceededError`を追加してHTTP 429で拒否する。
- **アイドルセッションタイムアウト**: `config.py`に`get_idle_timeout_seconds() -> float`（`IDLE_SESSION_TIMEOUT_SECONDS`環境変数、デフォルト`1800`＝30分）を追加。`SessionManager`は`session_id`ごとの最終アクティビティ時刻`_last_activity: dict[str, float]`を持ち、`create()`で初期化、`get()`と`is_stop_requested()`が呼ばれるたびに更新する。放置されたセッション（誰も`/snapshot`・`/command`・`/stop`を叩かず、`/run`の進行中ポーリングも無い＝定義上「孤児」）は以後一切リクエストを受けないため、リクエスト駆動の遅延チェックでは検知できない。したがって`SessionManager.__init__`で常駐のデーモンスレッド（`threading.Thread(daemon=True)`）を1本起動し、一定間隔（モジュール定数、環境変数化はしない）で全セッションを走査し、`idle_timeout_seconds`を超えたものを`close()`する。
- 上記のセッション上限チェック（数える→作る、の間の競合）とアイドルスイープの`close()`が同じ`_sessions`等の辞書を触るため、`SessionManager`に1本の`threading.Lock`を新設し、`create()`・`close()`（前景の`DELETE`経由と背景スイープ経由の両方）をこのロックで囲む。`get()`・`is_stop_requested()`等の読み取り系は引き続きロックしない（既存のStep2の「並行アクセス安全性は対象外」という割り切りの範囲内として残す）。この判断の理由は決定事項節参照。
- 動作確認は3本柱のうちURL許可リスト（`target_url`・`/command`両経路）とセッション上限・アイドルタイムアウトの3点は実AI APIコール無しで検証できる（`scripts/temp/test_server.py`と同種のAIフリーなHTTPスモークテスト）。`/run`中にAIのツール呼び出し（`navigate`やクリック起因）で許可リスト外に遷移するケースの確認だけは実AI APIコールを要するため、既存の`vertical-slice-ai-test`スキル・Step7の方針を踏襲し、実行前にユーザーへ確認する。

## 決定事項・注意点／落とし穴

| 決定 | 理由 |
|---|---|
| 許可リスト外への遷移は**即停止**（警告のみにはしない） | このシステムは`/run`においてAIが人間の介入なしに継続的にツール呼び出しを行う全自動ループであり、警告に留めるとAIが許可外ドメイン上で操作を続けてしまうリスクがある。SPEC.md 7章が安全性を明示的にスコープ外としつつも、big_plansが「実装コストが低いため最低限のガードだけ入れる」としている以上、コストの低い「即座に止める」を選び、追加の警告専用パイプラインは作らない |
| チェックの関門は`navigate`ツール／`target_url`単体ではなく`CliExecutor.execute()`内の1箇所（`Page URL:`のポストホック解析） | `execute()`は人間駆動（`/command`）・AI駆動（`tools.execute_tool`経由）・`open()`/`run_code()`のすべてが通る唯一の関門であり、しかもplaywright-cliは**あらゆる**コマンドの実行後にPage状態を返す。`navigate`ツール呼び出しだけをチェックする案は、クリックによる間接遷移（リンククリックがナビゲーションを引き起こす等）を確実に取りこぼす。1箇所に集約することで、`/command`と`/run`の両方に同じロジックを重複実装せずに済む |
| 遷移前のURL文字列を事前検証する経路は別途追加しない（`navigate`/`open`のように遷移先URLが呼び出し時点で分かっているケースでも） | 事前検証を足すと「既知URLの事前チェック」と「Page URLのポストホック検知」の2経路が併存し、この機能が避けたかった重複を自ら持ち込むことになる。ポストホック検知では対象ページへ一瞬遷移してしまうこと自体は許容する（SPEC.mdが認証必須サイト・破壊的操作の個別ガードを明示的にスコープ外としている以上、「一瞬読み込まれること」自体のリスクは本ステップの対象外） |
| `DisallowedNavigationError`は`CliError`のサブクラスとして定義する | 既存の`except CliError`（`step_runner.py`・`app.py`の3箇所）がそのまま拾えるため、伝播経路を壊さずに済む。区別したい箇所（`step_runner.py`のfailure_notes分岐、`app.py`のHTTPステータス分岐）だけ、より具体的な`except DisallowedNavigationError`をその前に追加する最小差分で対応する |
| `POST /sessions`の`target_url`違反・`/command`での違反はHTTP 400、セッション上限超過は429、それ以外の`CliError`は既存どおり502 | 400は「クライアントの要求そのものがポリシー上拒否された」ことを表し、502（CLIレベルの実行時失敗）と意味的に区別できる。429はセッション枯渇に対する標準的な意味（作れない・後で試せ）に合致する。新しいステータスコード体系を発明せず、REST的に定着した意味へ寄せる |
| AI駆動パス（`/run`・`/sessions/resume`）の違反は新HTTPステータスを設けず、既存の`failure_notes`パイプラインに`reason: "disallowed_url"`を追加するだけで表現する | Step6/Step7が確立した「`failure_notes`は`step`/`reason`/...の同じ形」という規約に5番目・6番目の`reason`（Step7の`stopped`に続く）として乗せるのが最小の変更。`/run`はHTTPリクエスト内で同期実行される既存の設計上、違反はレスポンスの成否（`passed: false`）と`failure_notes`で十分表現できる |
| `disallowed_url`はリトライ対象から除外し、`run_task_logged_step`の残り試行を消費せず即座に打ち切る | 許可されていないドメインに対して追加のAIツール呼び出しを行わせるのは「即停止」の趣旨に反する。Step7の`should_stop`早期脱出と同じ「打ち切り」の形を流用し、新しい打ち切り種別を表す独立フィールドは追加しない（`reason`自体が`disallowed_url`であることが既に「なぜ打ち切られたか」を表しているため、Step7の`"stopped": true`のような追加フラグは不要） |
| 違反発生時にセッション自体を自動closeする処理は追加しない（`POST /sessions`の`target_url`違反時の既存の`sessions.close(session_id)`は維持しつつ、`/run`・`/command`の違反では何もしない） | `runner.py`/`step_runner.py`は生の`CliExecutor`だけを受け取り`SessionManager`のハンドルを持たない設計になっており、そもそも構造的にセッションをcloseできない。人間がその場のブラウザ状態を`/snapshot`で確認したうえで`DELETE`するか、別セッションで`resume`するかを選べるようにするため、他の`cli_error`失敗と同じ「セッションは残す」扱いに揃える |
| `SessionManager`にセッション上限（`max_sessions`）を持たせ、`create()`の先頭（`CliExecutor`のプロセス起動より前）でチェックする | 起動してから拒否すると無駄にplaywright-cliプロセスを生成・破棄することになり、まさに今回防ぎたい「プロセスが積み上がる」リスクを部分的に自ら作ってしまう |
| セッション上限チェック（件数を数える→`CliExecutor`生成→登録）とアイドルスイープの`close()`を`threading.Lock`1本で囲む。`get()`・`is_stop_requested()`等の読み取り系はロックしない | 今回初めて「HTTPリクエストとは独立に動くバックグラウンドスレッド」がレジストリを書き換える。Step2以来の「並行アクセス安全性は対象外」という割り切りは「複数HTTPリクエストが同じセッションに同時に来るケース」を指しており、それとは質的に別の懸念（辞書の`del`が二重に走って`KeyError`になる、上限チェックが競合してカウントを超えて作られる、という致命的な破損）を防ぐのがこのロックの唯一の目的。読み取り系まで含めて全面的にロックすると、Step2が明示的に対象外とした「並行アクセスの安全性」まで肩代わりすることになり、スコープを不必要に広げてしまうため、破損を防ぐ最小限（作成・削除のみ）に留める |
| アイドルタイムアウトの検知は常駐のバックグラウンドスレッドによる定期スイープとし、リクエスト駆動の遅延チェック（アクセスの都度期限切れか確認する）にはしない | 「孤児セッション」は定義上、以後誰も`/snapshot`・`/command`・`/stop`・`DELETE`を叩かない（叩けば孤児ではない）ため、リクエスト駆動のチェックは本来検知したいケースに対して原理的に無力。能動的に閉じるにはリクエストとは独立したトリガーが必須で、このリポジトリでStep7が導入した`threading.Event`に続き2つ目のスレッド関連プリミティブ（デーモンスレッド）の導入になる |
| 最終アクティビティの更新は`SessionManager.get()`と`is_stop_requested()`の2箇所だけに限定する（`is_stop_requested`は「停止確認」と「アクティビティ更新」の二重責務を持つ） | `get()`は`/snapshot`・`/command`・`/stop`・`/run`開始時の`_get_cli`が必ず経由するため人間駆動の活動を自然にカバーする。`is_stop_requested()`は`runner.run_steps`/`run_task_logged_step`がステップ境界・リトライ境界ごとに呼ぶ（Step7で確立済みの頻度）ため、長時間実行中の`/run`にも活動記録を波及させられる。この二重責務化により`runner.py`/`step_runner.py`/`orchestrator.py`へ新しい“タッチ”専用の引数やコールバックを配線せずに済む（Step7が確立した`should_stop`のシグネチャを一切変えない） |
| 既知の限界: 1ステップの全ターン（最大`MAX_TURNS_PER_STEP`＝8ターン）が`idle_timeout_seconds`を超えて長引いた場合、その間`is_stop_requested()`はステップ内では呼ばれない（Step7同様、ターン単位のフックは無い）ため、進行中のステップの途中でスイープに閉じられてしまう可能性が理論上残る | `step_runner.run_step()`のターン制御ループには一切手を入れない、というStep4以降一貫した制約を今回も踏襲するため、ターン単位でアクティビティを更新するフックは追加しない。実運用では`idle_timeout_seconds`のデフォルト（30分）が1ステップの所要時間を大きく上回る想定であり、許容できるリスクとして受け入れる |
| セッション上限・アイドルタイムアウトのデフォルト値は常時有効（`ALLOWED_DOMAINS`と違い「未設定なら無制限」にはしない）とし、無効化のための明示的なスイッチも用意しない | プロセス・ブラウザリソースの積み上がり防止という目的自体はテスト対象のドメインに関わらず常に成り立つため、デフォルトで有効にしておくのが「閉じ忘れ対策」というbig_plansの意図に合う。無効化したい場合は大きな値を環境変数に設定すればよく、専用のON/OFFフラグはYAGNI（`MAX_STEP_ATTEMPTS`に環境変数オーバーライドを用意していない既存の判断と同じ姿勢） |
| ホスト比較はポート番号を無視する（`urlparse(...).hostname`、`netloc`ではない） | `scripts/stories/*.yaml`の`seed_url`が`http://localhost:8080`のようにポート付きである一方、`ALLOWED_DOMAINS`はテスト対象ドメインを指定する運用者向け設定として「ホスト名」の単位で書かせたい（`localhost:8080`と`localhost:3000`を別エントリとして毎回書かせるのは冗長）。破壊的操作の個別ガード等はスコープ外である以上、同一ホストの別ポートを区別する必要性も薄い |
| `ALLOWED_DOMAINS`だけは未設定時に「制限なし」をデフォルトにする（他の2つと非対称） | 既存のデモシナリオ（`scripts/stories/*.yaml`のほとんどが`localhost:8080`、1本が`playwright.dev`）が環境変数の追加設定なしに動き続けられることを優先する。許可リストは「何をテストするか」というテスト対象固有の設定であり、プロセスリソースの保護（常時有効であるべき他の2つ）とは性質が異なる。運用者が明示的に`ALLOWED_DOMAINS`を設定した時にだけガードが働く、というopt-inな設計にする |
| スイープの走査間隔はモジュール定数としてハードコードし、環境変数化しない | 検知の粒度（あるセッションは実際の`idle_timeout_seconds`より最大でこの間隔分だけ長く生き残りうる）という実装都合の値であり、運用者が調整すべきパラメータではない。既存の`MAX_STEP_ATTEMPTS`同様「小さく始めて後で調整」のYAGNI姿勢を踏襲する |
