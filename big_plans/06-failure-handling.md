# Step 6: 失敗時のふるまい

## 目的

SPEC.md 6章のリトライ・停止ロジックを実装する。

## やること

- タスクが失敗した場合、まず自動でリトライする（人間の代理操作はしない。コード化されず一時しのぎにしかならないため）
- リトライ回数の上限を決める（SPEC.md 9章 Open Question）。まずは小さい値（例: 2〜3回）から始めて運用しながら調整する
- 上限に達して停止する際、`playwright-cli` の `heal` フロー（[test-generation.md](../.claude/skills/playwright-cli/references/test-generation.md) 3章）に倣い、以下を診断情報として添えて人間に提示する
  - 失敗直前の snapshot / スクリーンショット
  - `console` / `requests` の内容（該当する場合）
  - 失敗の推定原因（セレクタ変化・想定外の画面遷移・タイムアウトなど）
- 「仕様が古いのか、アプリ側の回帰バグなのか」の判断はサーバー側では行わず、人間に確認を求める形にする（heal.md 3.4節と同じ思想）

## 完了条件

- 意図的に失敗するタスク（例: 存在しない要素をクリックさせる）を用意し、リトライ→停止→診断情報つき提示、の流れが動作することを確認する
- 診断情報を見た人間が、原因調査に `playwright-cli attach` 等で追加のCLI操作をしなくても、ある程度当たりをつけられる情報量になっているか確認する
