# Step 5: 記録と途中再開

## 目的

SPEC.md 3章の「各操作の記録が残っているため、任意の時点からの復元が可能」を実装する。

## やること

- 各タスクの実行時に以下を記録する
  - 実行した `playwright-cli` コマンドと生成コード
  - 実行前後のスクリーンショット
  - 実行前後の snapshot
  - タスクの成否
- 記録からの復元を実装する
  - 例: `state-save` / `state-load`（cookie・localStorage）や、記録済みの操作列を先頭から再実行することでブラウザの状態を任意のタスク開始時点まで戻す
- 「途中のタスクから再実行する」「別パターンのテストへ分岐する」という、README概要にある要件を実際に試す
  - 例: タスク3までは記録から復元し、タスク4以降を別の入力で実行する

## 完了条件

- 記録されたログだけを使って、途中のタスクからテストを再開できることを確認する
- 復元できないケースが見つかった場合は、SPEC.md 3章の「スコープ外」に該当するかどうかを判断し、該当しなければ設計を見直す

## 追加: AI呼び出しのモデル記録の正確性

### 背景

`scripts/internal/cost_summary.py`（コスト集計スクリプト）は `<out>.steps.jsonl` の `model` フィールドを元に `model_pricing.csv` を引いてコストを試算する。しかし現状 `step_runner.run_step()` がログへ書き込む `model` は呼び出し元が渡したリクエスト文字列（`AI_MODEL` 環境変数 / `--model` 引数）であり、OpenAI Responses APIのレスポンスが実際に解決したモデル名（`response.model`）ではない。エイリアス的なモデル名を指定した場合にリクエストとレスポンスのモデル名が一致するとは限らず、ログから「確実に使用されたモデル」を判断できない。

### やること

- `step_runner.run_step()` が `append_step_log()` に渡す `model` フィールドを、リクエストパラメータではなく `response.model`（レスポンスが実際にエコーバックする解決済みモデル名）に置き換える
- リクエスト時に指定したモデル文字列も別フィールド（例: `requested_model`）として残し、両者が食い違うケース（エイリアス解決・フォールバックなど）を後から確認できるようにする
- `cost_summary.py` のモデル解決・pricing lookup を、実際に使用が確認された `model`（＝`response.model`）優先で見直す

### 完了条件

- 新たに生成された `<out>.steps.jsonl` の各ターンで、`model` がAPIレスポンスの `response.model` と一致していることを確認する
- リクエスト時のモデル文字列とレスポンスの `model` が異なるケースを1つ再現し、`requested_model` と `model` の両方が正しく記録されることを確認する
- `cost_summary.py` が、この新しい記録を使って `--model` オーバーライド無しでも正しい価格行を引けることを確認する
