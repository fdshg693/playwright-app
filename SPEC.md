# SPEC

まだ検討段階。細かい構成・仕様は今後いくらでも変わってよい前提で記載する。

## 1. 全体アーキテクチャ

- **サーバー**: テストしたい画面をラップし、AIからのリクエストを受け付ける
  - Playwright CLI（`playwright-cli`）を子プロセスとして操作し、画面操作・ログ収集を行う
  - テスト実行中はブラウザセッション（`-s=<name>`）を1本、実行の最初から最後まで維持する
  - 各タスクの操作ログ・生成コード・スクリーンショットを記録する
  - 人間からの強制停止・状態確認のためのエントリポイントを持つ
- **AIエージェント**: 1タスク＝1回のフレッシュなコンテキストでのモデル呼び出し
  - サーバーから「残りのテストストーリー（次ステップ含む）」と「現在の画面情報（snapshot）」を受け取る
  - `playwright-cli` コマンド列を発行して画面を操作する
  - 過去のタスクでの操作履歴には（設計上）依存しない

### セッションの永続性とコンテキストの非永続性の違い

ここは分けて考える必要がある：

- **ブラウザセッション（Playwright CLIのセッション）** はテスト実行の間、サーバー側で永続する
- **AIのコンテキスト** はタスクごとに初期化される

サーバーの役割は、永続するセッションに対して都度 `snapshot` を取得し、その結果を新しいAIコンテキストに渡すことで、「フレッシュなコンテキストでも今の画面がわかる」状態を作ること。

## 2. タスク分割とコンテキスト設計

- 各タスクへの入力は以下の2つのみ
  1. 次のステップを含むテストストーリー
  2. 現在の画面概要（`playwright-cli snapshot` の結果など）
- 以降の操作は「今までの操作」に強く依存しない設計とする。現在の画面情報だけで判断が完結することを前提にする
- **スコープ外**: 現在の画面情報だけでは足りない（＝過去の操作結果を記憶していないと解けない）ようなテストシナリオは、そもそも対象外とする

### タスクの粒度: ステップ単位

**1タスク＝1ステップ**とする（シナリオ単位の1段階粗い粒度は採用しない）。

Playwright CLIの `plan → generate → heal` ワークフロー（[test-generation.md](.claude/skills/playwright-cli/references/test-generation.md)）は「1シナリオ＝1セッション」という粒度感を持っているが、これはCLIセッション（ブラウザ）の単位であって、AIのコンテキストを初期化する単位とは別に考える。ブラウザセッションはシナリオ単位で維持しつつ、AIへのタスク割り当てはステップ単位まで細かくする。

理由:

- ブラウザ操作は現時点でAIにとって本質的に複雑なタスクであるという前提に立つ。コンテキストが大きくなるほど、判断ミスや目的から外れた操作（横道にそれる）が発生しやすくなる
- 同じ画面情報を毎回渡すことによる初回オーバーヘッドは発生するが、それ以上に「コンテキスト増大に伴うミスの頻発」「横道にそれることによるリトライ・手戻り」のコストの方が大きい
- 結果として、トークン消費・実行時間の両面でも、粗い粒度より細かい粒度の方が抑えられると見込む
- この判断は「ブラウザ操作がAIにとって本質的に複雑」という前提に依存する。将来的にAIの精度が上がりこの前提が崩れる場合は、粒度を粗くする方向で見直す

## 3. セッション管理・途中再開

- サーバーがPlaywright経由でブラウザをヘッドありで起動する
- 各操作の記録（生成コード・snapshot・スクリーンショット）が残っているため、任意の時点からの復元が可能
- **スコープ外**: 記録から復元できないケース（例: 外部要因で状態が再現不能な場合）への対応は当面考慮しない

## 4. AIの操作インターフェース: Playwright CLI

- [Playwright MCP](https://github.com/microsoft/playwright-mcp) ではなく **Playwright CLI** を採用する
- 理由:
  - MCPは「AI → Playwright → 画面」が閉じたループで完結する分には楽だが、その中間に独自ロジック（タスク管理・記録・再開制御など）を挟もうとした瞬間、MCP実装の作法に合わせる必要がありアダプトコストが発生する
  - CLIは薄いプロセス呼び出しのインターフェースなので、簡便さを保ちながら独自ロジックを挟むコストを最小化できる
  - これは車輪の再発明ではなく、Playwright CLIが持つ機能（セッション管理・snapshot・コード生成・トレース記録等）をそのまま利用しつつ、その上に薄いオーケストレーション層を足す構成

## 5. テストコードの生成

- `playwright-cli` は各操作ごとに対応するPlaywright TypeScriptを自動生成する（role-basedロケータ優先）
- テスト生成は既存の `playwright-cli` の **plan → generate → heal** フローに乗せる
  - plan: アプリを探索し、シナリオ一覧を `specs/*.plan.md` に書き出す
  - generate: シナリオごとに操作を実行し、生成されたコードをテストファイルへ組み立てる
  - heal: 失敗したテストの原因を診断し、コード・spec を修正する
- 自作するのはこのフローを人間の代わりにAIが自動運用する部分（オーケストレーション）であり、コード生成そのものの品質保証機構は作らない

## 6. 失敗時のふるまい

- タスクが失敗した場合は **リトライ** する
  - 人間が代わりに操作しても、その操作はコード化されない（一時しのぎにしかならない）ため、人間による介入は基本的に行わない
- リトライしても失敗が続く場合は、そこで **停止** する
  - 停止時には `heal.md` の思想（セレクタ変化／アプリ側の想定外遷移／タイムアウトなど、何が原因で失敗したか）に倣い、診断情報を添えて人間に提示する
  - 「仕様が古いのか、アプリ側の回帰バグなのか」の判断が必要な場合は、判断せず人間に確認を求める

## 7. 安全性

- 現時点では **完全にスコープ外**
- 当面は認証不要のWebサイトのみを対象とする
- サーバー側でテスト対象のURLを固定することで、意図しない画面への遷移を防ぐ（実装コストは低い想定）

## 8. 人間による確認・介入

- 途中経過・最終結果を人間が確認できるようにする（具体的な確認手段はUI含めて要検討）
- 強制停止の手段を持つ
